## 1. Цели и задачи

### 1.1. Цель проекта

Разработка кроссплатформенного десктопного браузера на базе открытого движка Chromium, обеспечивающего полную совместимость с российскими государственными информационными системами, криптографическими стандартами (ГОСТ) и сервисами экосистемы «Чебурнета».

### 1.2. Задачи

| № | Задача | Приоритет | Срок |
|---|--------|-----------|------|
| 1 | Сборка базовой сборки Chromium с интеграцией GOST-криптографии | Критический | Этап 1–2 |
| 2 | Реализация криптографической подсистемы (ГОСТ Р 34.10-2012, ГОСТ Р 34.11-2012) | Критический | Этап 1–2 |
| 3 | Интеграция корневых сертификатов НУЦ Минцифры и модуля автоматического обновления | Высокий | Этап 2 |
| 4 | Реализация проксирования и DNS-резолвинга через российские DNS-серверы | Высокий | Этап 2–3 |
| 5 | Разработка UI/UX с нативной интеграцией VK ID / Госуслуг | Высокий | Этап 3 |
| 6 | Предустановленный пакет закладок и виджетов сервисов | Средний | Этап 3 |
| 7 | Модуль блокировки рекламы, трекеров и аналитических скриптов | Средний | Этап 3–4 |
| 8 | Тестирование национальных стандартов кодирования и специфических технологий | Высокий | Этап 4 |
| 9 | Подготовка к сертификации (ФСБ, ФСТЭК) | Критический | Этап 4–5 |
| 10 | Публикация и распространение | Средний | Этап 5 |

### 1.3. Целевая аудитория

- Государственные учреждения и органы власти
- Критическая инфраструктура и.enterprise-сегмент
- Образовательные и научные организации
- Пользователи, заинтересованные в приватности и суверенном ПО

### 1.4. Ожидаемые результаты

- Рабочий десктопный браузер для Windows, Linux, macOS
- Полная поддержка ГОСТ-криптографии для TLS-соединений

- Встроенная блокировка рекламы и трекеров
- Нативная интеграция с VK ID и ЕСИА (Госуслуги)

#### 2.2.3. Интеграция GOST-модуля

Интеграция осуществляется через патчи к BoringSSL и добавление собственного криптографического модуля:

```
src/
├── third_party/
│   └── gost_engine/              # GOST-движок для OpenSSL/BoringSSL
│       ├── CMakeLists.txt
│       ├── gost_engine.c         # Точка входа движка
│       ├── gost_grasshopper.c    # ГОСТ Кузнеца (шифрование)
│       ├── gost_gost89.c         # ГОСТ 28147-89
│       ├── gost_grasshopper.h
│       └── ...
├── chrome/browser/gost/          # Браузерная интеграция
│   ├── gost_tls_provider.cc
│   ├── gost_certificate_verifier.cc
│   └── gost_prefs_bridge.cc
└── patches/
    ├── chromium_gost.patch       # Патч к BoringSSL
    ├── chromium_net.patch        # Патч к networking stack
    └── chromium_ui.patch         # Патч к UI-слою
```

#### 2.2.4. Скрипт сборки

```bash
#!/bin/bash
# scripts/build.sh — Основной скрипт сборки

set -euo pipefail

PLATFORM="${1:-linux}"
CONFIG="${2:-Release}"
JOBS=$(nproc 2>/dev/null || sysctl -n hw.ncpu 2>/dev/null || echo 4)

echo "=== Сборка Чебурнет-браузера ==="
echo "Платформа: $PLATFORM | Конфигурация: $CONFIG | Параллелизм: $JOBS"

# 1. Применение патчей
echo "[1/6] Применение патчей..."
cd src
git apply --check ../patches/*.patch
git apply ../patches/*.patch

# 2. Генерация ninja-файлов
echo "[2/6] Генерация build-системы..."
gn gen "out/$CONFIG" --args=''"$(cat "../configs/${PLATFORM}_${CONFIG}.gn")"'"

# 3. Сборка зависимостей
echo "[3/6] Сборка зависимостей..."
ninja -C "out/$CONFIG" -j"$JOBS" third_party/blink/renderer/platform

# 4. Сборка криптографического модуля
echo "[4/6] Сборка GOST-модуля..."
ninja -C "out/$CONFIG" -j"$JOBS" gost_engine

# 5. Полная сборка
echo "[5/6] Полная сборка браузера..."
ninja -C "out/$CONFIG" -j"$JOBS" chrome

# 6. Сборка инсталлятора
echo "[6/6] Сборка инсталлятора..."
ninja -C "out/$CONFIG" -j"$JOBS" chrome_installer

echo "=== Сборка завершена ==="
echo "Результат: out/$CONFIG/"
```

## 3. Функциональные требования

### 3.1. Криптографическая подсистема

#### 3.1.1. Поддерживаемые алгоритмы

| Стандарт | Назначение | Алгоритм |
|----------|-----------|----------|
| ГОСТ Р 34.12-2015 «Кузнечик» | Симметричное шифрование (AES-аналог) | 128/256-битный ключ |
| ГОСТ Р 34.13-2015 | Режимы блочного шифрования | CTR, CBC, OMAC |
| ГОСТ Р 34.10-2012 | Электронная подпись (ECDSA-аналог) | 256/512-бит |
| ГОСТ Р 34.11-2012 «Стрибог» | Хэш-функция (SHA-аналог) | 256/512-бит |
| ГОСТ Р 34.10-2012 + Р 34.11-2012 | Обмен ключами (ECDH-аналог) | 256/512-бит |
| RFC 9578 / 8998 / 9045 | TLS 1.3 с GOST-шифрами | Кузнечик + Стрибог |

#### 3.1.2. Архитектура криптографической подсистемы

```
┌─────────────────────────────────────────────────────┐
│                   Браузер (UI)                       │
├─────────────────────────────────────────────────────┤
│              Network Layer (net/)                    │
│         ┌───────────────────────────────┐           │
│         │     TLS 1.3 Implementation    │           │
│         │       (BoringSSL)             │           │
│         └──────────┬────────────────────┘           │
│                    │                                 │
│         ┌──────────▼────────────────────┐           │
│         │   Crypto Provider Interface   │           │
│         │     (NSS / BoringSSL hooks)   │           │
│         └──────────┬────────────────────┘           │
│                    │                                 │
├────────────────────┼────────────────────────────────┤
│         ┌──────────▼────────────────────┐           │
│         │     GOST Crypto Engine        │           │
│         │  ┌──────────┐ ┌────────────┐  │           │
│         │  │Кузнечик  │ │  Стрибог   │  │           │
│         │  │(шифр.)   │ │  (хэш)     │  │           │
│         │  └──────────┘ └────────────┘  │           │
│         │  ┌──────────┐ ┌────────────┐  │           │
│         │  │ГОСТ 34.10│ │Обмен ключ. │  │           │
│         │  │(подпись) │ │(key agree) │  │           │
│         │  └──────────┘ └────────────┘  │           │
│         └───────────────────────────────┘           │
│              Опциональный HSM/Token                  │
└─────────────────────────────────────────────────────┘
```

#### 3.1.3. Порядок выбора шифрового набора

1. При установке TLS-соединения клиент отправляет `supported_groups` с GOST-группами
2. Если сервер поддерживает GOST — выбирается GOST-шифровой набор (prefixed с `0xC1`-диапазона)
3. При отсутствии поддержки сервером — откат на стандартные шифры (AES-GCM, ChaCha20)
4. Приоритет: GOST > AES-256-GCM > ChaCha20-Poly1305 > AES-128-GCM

#### 3.1.4. Требования к реализации

- **Производительность:** GOST-шифрование не должно снижать пропускную способность более чем на 15% по сравнению с AES-GCM
- **Реализация:** Движок Кузнечик должен использовать AES-NI-подобные инструкции (опционально, через platform-specific ускорение)
- **FIPS 140-3:** Криптографический модуль должен соответствовать требованиям ФСБ ТК-26 (Техническое задание)
- **Резервный путь:** При недоступности GOST-модуля браузер должен работать в режиме совместимости с обычными шифрами

#### 3.4.1. Архитектура блокировщика

#### 3.4.2. Правила блокировки

| Категория | Источники правил | Приоритет |
|-----------|-----------------|----------|
| **Рекламные сети** | EasyList, RuAdList, Fanboy's Annoyances | Высокий |
| **Трекеры** | EasyPrivacy, I don't care about cookies | Высокий |
| **Аналитика** | Кастомные правила (Яндекс.Метрика, Mail.ru/Rating@Mail.ru) | Средний |
| **Криптомайнеры** | Криптомайнер-листы | Критический |
| **Социальные виджеты** | Fanboy's Social | Средний |
| **Фишинг** | Фильтры НУЦ (опционально) | Высокий |

#### 3.4.3. Реализация фильтрации

- **Реализация:** uBlock Origin-совместимый движок ( declarativeNetRequest + content scripts )
- **Производительность:** Фильтрация на уровне networking stack, не на уровне рендеринга
- **Кэширование:** Отфильтрованные правила кэшируются в памяти
- **Обновление:** Автоматическое обновление фильтр-листов раз в 4 часа
- **Пользовательские правила:** Поддержка пользовательских фильтров (Syntax: ABP)
- **Отладка:** `chrome://adblocker` — страница диагностики с логами блокировок

#### 3.4.4. Конфиденциальность

| Механизм | Описание |
|----------|----------|
| **Do Not Track** | Отправка заголовка DNT при включении |
| **Global Privacy Control** | Поддержка GPC |
| **Трекинг** | Блокировка отслеживания fingerprinting |
| **Синхронизация** | Шифрование synced-данных на стороне клиента |
| **Отчёты** | Анонимная телеметрия (опционально, по умолчанию выключена) |
| **Политика** | Минимальный сбор телеметрии, прозрачная политика |

## 4. Требования к безопасности

### 4.1. Криптографическая безопасность

#### 4.1.1. Требования к TLS

| Параметр | Минимум | Рекомендуемый |
|----------|---------|---------------|
| Версия протокола | TLS 1.2 | TLS 1.3 |
| GOST-шифры | Кузнечик-CTR + MAC | Кузнечик-OMAC (AEAD) |
| Обмен ключами | ГОСТ 34.10-2012 (256) | ГОСТ 34.10-2012 (512) |
| Хэш-функция | Стрибог-256 | Стрибог-512 |
| OCSP | Обязательно | Must-Staple |
| CT (Certificate Transparency) | Поддержка | Активная проверка |

#### 4.1.2. Управление ключами

- Приватные ключи хранятся в системном хранилище (DPAPI / Keychain / keyring)
- Поддержка аппаратных токенов (Рутокен, eToken) через PKCS#11
- Минимальная длина ключа: 256 бит (ГОСТ 34.10-2012)
- Ротация ключей: автоматическая, раз в 365 дней

### 4.2. Сертификатная подсистема

#### 4.2.1. Доверенные корневые сертификаты НУЦ Минцифры

| Параметр | Описание |
|----------|----------|
| **Источник** | https://gu.noc.ru/ или официальный сайт НУЦ |
| **Формат** | DER → PEM конвертация, интеграция в trust store |
| **Хранение** | Отдельный trust store с маппингом на системный |
| **Корневые** | Минимум: РУЦ-ГНУЦ-ГТС, РУЦ-ИГА, РУЦ-ФНС, РУЦ-ПФР и др. |
| **Промежуточные** | Динамическая загрузка по AIA (Authority Information Access) |

#### 4.2.2. Автоматическое обновление сертификатов

| Компонент | Описание |
|-----------|----------|
| **Механизм** | Собственный CRL/OCSP проверочник с кэшированием |
| **Интервал обновления** | Корневые: раз в 30 дней, CRL: раз в 4 часа |
| **Fallback** | При недоступности НУЦ — локальная проверка по кэшу |
| **Уведомления** | UI-уведомление при истечении/отзыве сертификата |
| **Совместимость** | Одновременная работа с системным trust store |
| **Логирование** | Журнал проверок в `chrome://certificate-log` |

#### 4.2.3. Процесс интеграции

```
1. Скачивание корневых сертификатов НУЦ
     ↓
2. Конвертация DER → PEM
     ↓
3. Проверка подписей (цепочка → корень)
     ↓
4. Добавление в Chromium trust store (cert_verify_proc)
     ↓
5. Патчинг cert_verifier.cc
     ↓
6. Настройка OCSP/CRL эндпоинтов
     ↓
7. Тестирование соединения с demo-сервером НУЦ
```

### 4.3. Защита от уязвимостей

| Механизм | Реализация |
|----------|-----------|
| **Sandbox** | Chromium sandbox (уже встроена) |
| **Сайт-изоляция** | Site Isolation (включена по умолчанию) |
| **ASLR** | Windows ASLR / Linux PIE |
| **DEP/NX** | Включена на всех платформах |
| **Control Flow Guard** | Включена на Windows |
| **Самообновление** | Signed updates, проверка по Ed25519 |
| **Откат версий** | Поддержка отката на предыдущую стабильную версию |

### 4.4. Аудит и сертификация

| Стандарт | Статус | Описание |
|----------|--------|----------|
| **ФСБ ТК-26** | Требуется | Требования к криптографическим средствам |
| **ФСТЭК** | Требуется | Требования к защите информации |
| **ФСБ (ГОСТ-шифры)** | Требуется | Соответствие ГОСТ Р 34.12-2015 |
| **PCI DSS** | Желательно | Для работы с платёжными системами |
| **ISO 27001** | Желательно | Управление информационной безопасностью |
